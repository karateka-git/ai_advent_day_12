# day_12 — Персонализация ассистента поверх layered memory

`day_12` продолжает [day_11](C:/Users/compadre/Downloads/Projects/AiAdvent/day_11).

Базовая layered memory model, pending-flow, short-term стратегии и общий memory pipeline уже были разработаны раньше. За подробным описанием этой части лучше смотреть [README day_11](C:/Users/compadre/Downloads/Projects/AiAdvent/day_11/README.md).

Этот этап сосредоточен уже не на самой memory model, а на **персонализации ассистента** поверх неё.

## Что сделано в `day_12`

### 1. Профиль пользователя встроен в `long-term`

Профиль не хранится отдельным top-level объектом.

Вместо этого:

- у `long-term` заметок появился владелец;
- заметки могут быть `global` или `user-scoped`;
- профиль пользователя считается представлением user-scoped части `long-term`.

Это позволило не дублировать память и не плодить отдельный storage для профилей.

### 2. Поддержано несколько пользователей

Теперь в состоянии памяти есть:

- список пользователей;
- `activeUserId`;
- user-scoped заметки для конкретного пользователя.

Смена профиля работает через переключение активного пользователя:

- `/users`
- `/user`
- `/user create <id> [display_name]`
- `/user use <id>`

При этом:

- user-scoped память переключается вместе с активным пользователем;
- `global` память остаётся общей;
- новые профильные заметки сохраняются уже для текущего активного пользователя.

### 3. Профиль автоматически подмешивается в каждый запрос

Итоговый prompt теперь собирается так:

1. system prompt
2. профиль активного пользователя
3. общий `long-term`
4. `working`
5. `short-term`

Профиль попадает в prompt не как сырой список заметок, а как отдельный блок правил ответа:

- с явным приоритетом относительно стандартного поведения ассистента;
- с инструкцией применять профиль автоматически;
- с указанием, что профиль важнее инерции предыдущих ответов в short-term диалоге.

Это нужно, чтобы персонализация влияла на ответы не только формально, но и заметно.

### 4. Добавлено управление профилем из CLI

Появились отдельные команды для работы с профилем:

- `/profile`
- `/profile add`
- `/profile add <category> <text>`
- `/profile note edit <id> text|category <value>`
- `/profile note delete <id>`

То есть персонализация теперь управляется не только через автоматическое извлечение, но и вручную.

### 5. Добавлено пошаговое добавление заметок

CLI теперь поддерживает пошаговый flow:

- `/memory add`
- `/profile add`
- `/cancel`

Это позволяет добавлять заметки без запоминания полной сигнатуры команды.

Flow работает как мастер:

- выбор слоя;
- выбор категории;
- ввод текста;
- подтверждение или отмена.

### 6. Снято ограничение “одна заметка на категорию”

Категория больше не считается уникальным ключом.

Теперь:

- в одной категории можно хранить несколько заметок;
- редактирование и удаление делаются по `id`;
- merge policy защищает от точных дублей, но не заменяет старую заметку новой только из-за совпадения категории.

Это особенно важно для профиля, потому что пользовательские предпочтения часто состоят из нескольких независимых правил внутри одной темы.

### 7. Rule-based allocator убран из рабочего контура

Автоматическое извлечение durable memory теперь строится вокруг `LlmMemoryLayerAllocator`.

Если utility LLM недоступен или allocator падает:

- используется `NoOpMemoryLayerAllocator`;
- новые кандидаты автоматически не извлекаются;
- ручное управление памятью и профилем остаётся доступным.

Такое поведение оказалось чище, чем поддержка второй, отдельной rule-based семантики.

### 8. Улучшена диагностика allocator-а

Trace allocator-а остаётся в:

- [build/logs/llm-memory-layer-allocator.log](C:/Users/compadre/Downloads/Projects/AiAdvent/day_12/build/logs/llm-memory-layer-allocator.log)

Это помогает разбирать:

- что именно ушло в utility LLM;
- что модель вернула;
- почему появился или не появился кандидат в память;
- был ли fallback в `NoOpMemoryLayerAllocator`.

## Как изменился memory pipeline

Базовый pipeline layered memory остался тем же, что был в `day_11`. В `day_12` изменились именно смысл и распределение durable memory.

### Что изменилось в durable memory

После пользовательского сообщения система по-прежнему:

1. добавляет сообщение в `short-term`;
2. запускает allocator для `working` и `long-term`;
3. валидирует кандидатов;
4. делит их на auto-apply и pending;
5. обновляет состояние памяти.

Но поверх этого появились новые правила:

- `long-term` теперь разделяется на `global` и `user-scoped` заметки;
- allocator должен учитывать владельца заметки;
- профиль активного пользователя собирается из user-scoped `long-term`;
- assembled prompt теперь включает отдельный профильный блок;
- ручное добавление заметок и профиля поддерживает пошаговый flow;
- category merge больше не уничтожает старые заметки той же категории.

### Что это изменило в поведении

Практический эффект такой:

- один и тот же проект теперь может обслуживать нескольких пользователей;
- ответы меняются в зависимости от `activeUserId`;
- профиль применяется автоматически, без ручного копирования инструкций в каждый запрос;
- проектная память и пользовательская память перестали смешиваться;
- память стала удобнее редактировать вручную;
- несколько предпочтений одной категории теперь могут сосуществовать, а не перезаписывать друг друга.

## Что проверено

В проекте дополнительно проверено:

- что профиль активного пользователя попадает в assembled prompt;
- что заметки другого пользователя не подмешиваются в ответ;
- что short-term не теряет ответы ассистента после refresh;
- что пошаговое добавление работает для памяти и профиля;
- что в одной категории можно хранить несколько заметок;
- что multi-user модель корректно переключает профиль активного пользователя;
- что проект проходит `.\gradlew.bat test`.
