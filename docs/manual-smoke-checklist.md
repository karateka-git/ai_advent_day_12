# Manual Smoke Checklist

Ручной smoke-check проекта `ai_advent_day_11` без comparison runner.

Цель:
- быстро проверить, что основное приложение запускается;
- CLI-команды работают;
- все пользовательские стратегии памяти работают;
- persisted state сохраняется в ожидаемой структуре.

Не входит в этот сценарий:
- `compareStrategies`
- judge
- dev-only comparison workflow

## Подготовка

1. Очистить runtime-файлы в `config/conversations/`.
2. Собрать и запустить проект:

```powershell
.\gradlew.bat build
.\gradlew.bat installDist
.\build\install\ai_advent_day_11\bin\ai_advent_day_11.bat
```

3. После каждого сценария при необходимости смотреть JSON в:

```text
config/conversations/
```

### Scripted-прогон с корректной UTF-8 кодировкой

Если нужно прогонять сценарий не вручную, а из заранее подготовленного файла команд, использовать helper-скрипт:

```powershell
.\scripts\run-scripted-session.ps1 -ScenarioFile .\path\to\scenario.txt -ClearConversations
```

Он:
- принудительно запускает приложение в UTF-8 режиме;
- корректно читает сценарий в UTF-8;
- сохраняет вывод в UTF-8 файл внутри `build/smoke-check/`.

## Базовый запуск

### Что проверить

- приложение стартует без ошибок;
- выводится стартовое сообщение;
- по умолчанию используется стратегия `Без сжатия`;
- команда `models` показывает список моделей;
- команда `use <id>` переключает модель и предлагает заново выбрать стратегию памяти.

### Минимальные действия

1. Запустить приложение.
2. Выполнить:

```text
models
```

3. Выполнить:

```text
clear
```

4. Отправить обычное сообщение:

```text
Привет, коротко опиши свои возможности.
```

### Ожидаемый результат

- приложение отвечает;
- token preview и ответ отображаются корректно;
- `clear` очищает контекст, не ломая сессию.

## Стратегия `Без сжатия`

### Действия

1. Если нужна смена модели, выполнить:

```text
use <model_id>
```

2. При выборе стратегии указать `Без сжатия`.
3. Отправить по очереди:

```text
Мы собираем ТЗ на Telegram-бота.
Срок запуска две недели.
Интеграция только с Google Sheets.
Напомни ограничения проекта.
```

### Ожидаемый результат

- ответ содержит оба факта:
  - две недели;
  - только Google Sheets;
- в JSON есть только `messages`;
- `strategyState` отсутствует или равен `null`.

## Стратегия `Сжатие через summary`

### Действия

1. Переключить модель через `use <model_id>`.
2. Выбрать `Сжатие через summary`.
3. Отправить не меньше 6 сообщений:

```text
Проект — бот для школы английского.
Цель — запись на пробный урок.
Бот должен работать только на русском.
CRM на первом этапе не нужна.
Бюджет до 120 тысяч рублей.
Собери краткий список ограничений.
```

### Ожидаемый результат

- ответы учитывают ранние факты из диалога;
- в JSON `strategyState` имеет тип `summary_compression`;
- внутри есть только summary-данные:
  - `summary`
  - `coveredMessagesCount`
- нет facts- или branching-полей.

## Стратегия `Скользящее окно`

### Действия

1. Переключить модель через `use <model_id>`.
2. Выбрать `Скользящее окно`.
3. Отправить:

```text
Первый факт: бюджет до 120 тысяч.
Второй факт: запуск за 2 недели.
Третий факт: только Telegram и Google Sheets.
Повтори только то, что осталось в коротком контексте.
```

### Ожидаемый результат

- ответ сильнее зависит от последних сообщений, чем от ранних;
- в JSON сохраняются все `messages`;
- `strategyState` отсутствует или равен `null`.

## Стратегия `Sticky Facts`

### Действия

1. Переключить модель через `use <model_id>`.
2. Выбрать `Sticky Facts`.
3. Отправить минимум 3 пользовательских сообщения с явными фактами:

```text
Цель проекта — записывать учеников на пробный урок.
Ограничение — без голосовых сообщений и без CRM.
Срок — две недели, бюджет — до 120 тысяч рублей.
```

4. Затем спросить:

```text
Сформулируй текущие ограничения и срок проекта.
```

### Ожидаемый результат

- ответ опирается на сохранённые facts;
- в JSON `strategyState` имеет тип `sticky_facts`;
- внутри есть:
  - `facts`
  - `coveredMessagesCount`
- нет summary- или branching-полей.

## Стратегия `Ветки диалога`

### Действия

1. Переключить модель через `use <model_id>`.
2. Выбрать `Ветки диалога`.
3. Отправить общую часть:

```text
Собираем ТЗ для Telegram-бота.
Нужен MVP за 2 недели.
Интеграция только с Google Sheets.
```

4. Создать checkpoint:

```text
checkpoint base_tz
```

5. Создать ветки:

```text
branch create mvp_light
branch create analytics_first
```

6. Перейти в первую ветку:

```text
branch use mvp_light
Убираем аналитику из первой версии.
Сформулируй краткое ТЗ для этой ветки.
```

7. Перейти во вторую ветку:

```text
branch use analytics_first
Добавляем аналитику заявок уже в MVP.
Сформулируй краткое ТЗ для этой ветки.
```

8. Проверить список веток:

```text
branches
```

### Ожидаемый результат

- ветки не смешивают контекст;
- `branches` показывает активную ветку и список всех веток;
- в JSON `strategyState` имеет тип `branching`;
- внутри есть только branching-данные:
  - `activeBranchName`
  - `latestCheckpointName`
  - `checkpoints`
  - `branches`

## Проверка persisted state

После прогонов по стратегиям проверить:

1. В корне состояния есть:
- `messages`
- `strategyState`

2. Для stateful стратегий `strategyState` сериализуется как отдельный тип:
- `summary_compression`
- `sticky_facts`
- `branching`

3. Для простых стратегий:
- `no_compression`
- `sliding_window`

`strategyState` отсутствует или равен `null`.

4. Нет старого общего metadata-слоя.

## Критерий успешного smoke-check

Smoke-check считается успешным, если:
- приложение запускается;
- базовые команды CLI работают;
- все 5 стратегий проходят свои сценарии без ошибок;
- persisted JSON соответствует ожидаемой структуре;
- ветки диалога действительно независимы.
